## Major Fixes Already Landed

### Claude Project Context

The claude.ai project context was migrated by adding `CLAUDE.md` at the repo
root. This is now the main long-term project context file for future agents.

### `complement NP` Gate Failure

Problem: `complement NP` failed in Level 3 with "The tactic 'NP' is not
available in this game!" because lean4game's gate checks the bare `NP` token
as if it were a tactic.

Fix: Level 3 introduces the phrase-type tokens with `NewHiddenTactic «NP» ...`
so they pass the tactic gate, while `NewDefinition` handles player-facing
vocabulary entries.

### Player-Side Target-String Checking

Problem: A player could solve Level 3 with arbitrary words because the level
only required a `DP`, not a `DP` whose surface yield was `my house`.

Rejected approach: development-side `#guard yield solution == "..."` checks
only the author's fixed sample tree and does not constrain browser players.

Final approach:

- `XSyntax.Tactics` defines `Utters b c s := { t : XTree b c // yield t = s }`.
- Each player tactic starts with `enterUtters` and ends with `closeUtters`.
- `enterUtters` turns an `Utters` goal into a visible tree goal plus a hidden
  yield proof obligation.
- `closeUtters` closes the hidden proof once the tree is complete, or reports
  a player-facing error such as:
  `✗ 这棵树念作 "the house",不是目标 "my house"`.
- Players do not type an extra `pronounce` line.
- Level statements now use targets such as
  `XSyntax.Utters .two .D "my house"`.
- A delaborator displays these goals as phrase plus target text, e.g.
  `DP：my house`.

### Multi-Word `head` Loophole

Problem: A player could pass target-yield checking by entering a full phrase in
one head, such as `head "my house"`, then using `head ""` for the complement.

Fix: `head` now accepts either one whitespace-free word or the explicit null
head `""`. Multi-word strings are rejected immediately:

`✗ head 一次只能种一个词；多个词必须各自占一个 head，空头才写 head ""`

This is not full lexical category checking; it only enforces "one head, one
word" plus explicit null heads.

### Definitions Panel Disappearing

Problem: after introducing hidden phrase-type tactics, the Definitions panel
lost `NP`, `VP`, etc.

Root cause: GameServer pools hidden names into one type-blind global set. A
hidden tactic named `NP` also hid a definition named `NP`.

Fix: keep the tactic-gate names bare (`NP`, `VP`, ...), but register
player-facing definitions under distinct bookkeeping keys (`NPDef`, `VPDef`,
...), with display names still set to `"NP"`, `"VP"`, etc. The player never
sees the `*Def` keys.

## Teaching/Game Design Diagnosis

The conceptual core is strong: grammaticality is modeled as tree existence,
with invalid structures made unrepresentable. The main weaknesses are in
sequence and player agency.

High-priority design concerns:

- Difficulty jumps sharply: L01=1 move, L02=3, L03=6, L04=11, L05 about 30.
- L05 introduces `specifier`, `adjoinR`, and null heads all at once.
- Hints mostly dictate exact commands, so players rarely decide between
  complement and adjunct themselves.
- Transfer is not tested: no unguided novel sentence such as `the big dog`.
- The best lesson, "ungrammatical means unbuildable", is mostly optional.
- The canonical transitive-verb complement example is missing, despite
  `Selects .V .D` existing.
- DP hypothesis, subject in Spec-TP, null C/T/D, and no movement are assumed
  silently; they should be disclosed pedagogically.
- The game shows grammatical-but-semantically-odd colorless examples, but not
  the converse meaningful-but-ungrammatical contrast.
- The reward is still mostly a type, not a visible tree.

Suggested teaching roadmap, not yet implemented:

- Insert a transitive VP level using `VD`.
- Give `specifier` its own level before the finale.
- Give `adjoinR` its own small level before the finale.
- Add a required ungrammatical-construction failure level.
- Add at least one unguided decision/transfer level.
- Localize remaining error messages and document theory commitments.
- Build the display/tree visualization layer.
